**Issue #140 — "Manual routing hit test"**
*Filed Jun 11 by upsuper*
Requests a custom callback that translates hit-test coordinates `(x, y)` to element-relative `(x', y')`. Current behavior uses the element's CSS-styled area for pointer dispatch regardless of canvas transformations/distortion, causing incorrect `:hover`, cursor, and event targeting.

**Issue #139 — "No escape hatch when drawElementImage mis‑renders content"**
*Filed Jun 11 by vanceingalls*
Documents hard failures with CSS 3D transforms, animated partial opacity, and accelerated canvases in `drawElementImage`. Argues that two restrictions compose badly: (1) `drawElementImage` only accepts immediate children, and (2) `layoutsubtree` children never paint to screen — so fallback screenshots return blank. Three asks: allow subtree capture, spec a `captureToImageBitmap()`-style fallback, or enumerate required CSS features so implementations throw rather than silently produce wrong pixels.

**Issue #138 — "Feature request: rasterise at a specified scale / devicePixelRatio"**
*Filed Jun 10, last updated Jun 11 by vanceingalls*
`drawElementImage` always rasterises at CSS pixel density with no way to supersample. On high-DPI displays or when output is larger than CSS size, text and vectors are visibly soft. Proposes an options bag (`{ scale: 2 }`) or honouring the destination canvas's backing-store density.
