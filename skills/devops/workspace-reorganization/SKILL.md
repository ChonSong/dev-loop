# Workspace Reorganization

Reorganize a cluttered workspace into topic-based directories without breaking existing script references.

## Core Principle

**Symlinks are debt, not cleanup.** Every symlink kept at root is a future maintenance burden. Create them only when an active script actually references the old path — and audit them after the move to remove dead ones.

## Phases

### Phase 1: Dependency Trace (BEFORE any moves)

1. List all root-level files
2. For each subdirectory with active scripts (`scripts/`, `agent-ops/`, `codi/`, etc.), grep for references to root-level filenames
3. Classify each root file:
   - **KEEP** — referenced by active scripts (e.g., `AGENTS.md`, `MEMORY.md`, `discovered-repos.json`)
   - **MOVE + SYMLINK** — referenced but can be moved (e.g., cron job targets)
   - **MOVE FREELY** — no references found

```python
# Pattern: find root files referenced from subdirs
import re
from pathlib import Path

root_files = {f.name for f in workspace.iterdir() if f.is_file()}
referenced = set()
for subdir in active_subdirs:
    for f in (workspace / subdir).rglob('*'):
        if f.is_file() and f.suffix in ('.py','.sh','.js','.json','.md'):
            content = f.read_text(errors='ignore')
            for name in root_files:
                if re.search(r'["\'][^"\']*' + re.escape(name) + r'["\']', content):
                    referenced.add(name)
```

### Phase 2: Create Topic Directories

```bash
mkdir -p education linkedin knowledge-graph screenshots archive infrastructure
```

### Phase 3: Move Files in Dependency Order

1. **Orphaned files first** (no references) — zero risk
2. **Output files** (generated by scripts) — update script output paths instead of symlinking
3. **Referenced files last** — move + create symlink at old path

### Phase 4: Symlink Audit (CRITICAL)

After ALL moves, verify each symlink is actually needed by an active script. Remove dead ones.

```python
for symlink in workspace.iterdir():
    if symlink.is_symlink():
        referenced = any(
            symlink.name in f.read_text(errors='ignore')
            for subdir in active_subdirs
            for f in (workspace / subdir).rglob('*')
            if f.is_file()
        )
        if not referenced:
            symlink.unlink()  # remove dead symlink
```

**This is where most reorganizations stop too early.** You end up with 40+ symlinks that nothing references. Always do this second audit pass.

### Phase 5: Update Script References

For moved files that are OUTPUT targets (scripts write TO them), update the script's output path:

```python
# Before: doc.save('/workspace/TEAC5019_Portfolio.docx')
# After:  doc.save('/workspace/education/TEAC5019_Portfolio.docx')
```

For INPUT targets, either symlink (third-party/unmodifiable) or update the path (scripts you control).

### Phase 6: Final Root Audit

Root should contain ONLY:
- Core agent/workspace docs (`AGENTS.md`, `SOUL.md`, `USER.md`, `MEMORY.md`, `HEARTBEAT.md`)
- Files actively referenced by scripts in subdirectories
- System config files
- Reorganization log

Target: **≤ 15 real files at root** for a typical workspace.

## Common Pitfalls

- **Creating symlinks preemptively** — don't symlink until you've confirmed an active script needs it
- **Moving files scripts write TO** — update the script's output path, don't symlink outputs
- **Forgetting to check all subdirectories** — `scripts/`, `agent-ops/`, `codi/`, `skills/`, `memory/`, archived dirs may all reference root files
- **Leaving dead symlinks** — after moving, re-scan and remove symlinks nothing references
- **Moving `.git/` directories** — never move or symlink `.git` dirs; repos with their own `.git` are self-contained
- **`git add -A` over-staging** — during a reorganization, `git add -A` stages EVERYTHING including subdirectories that should remain independent (embedded git repos, skill libraries, etc.). Stage only the root files and the `.gitignore` explicitly. Use `git reset HEAD` if you accidentally over-stage, then re-add selectively. Verify with `git status` before committing.
- **Embedded git repos absorbed into parent** — directories that have their own `.git/` are independent repos. When the parent workspace is a git repo, `git add -r <dir>` will try to add them as regular files, not submodules. Always add embedded repos to the parent `.gitignore` immediately to prevent accidental absorption. Pattern: `agent-ops/`, `gto-wizard-clone/`, `codi/`, `skills/`, `memory/` etc.
- **Permission boundary on root-owned files** — files/dirs owned by root (e.g. backup dumps like `onetag.bak/`) cannot be moved from an unprivileged container session. Don't fight it — note the move as a manual action item for the user. Never use `sudo` from the container (it's not available); use SSH to host or flag for manual action.
- **`cp -r` hanging on large directories** — `cp -r` on directories containing `node_modules/` or deep tree structures can hang or take minutes. Move files selectively (exclude `node_modules/`, `screenshots/`, `venv/`) or use `find -maxdepth 1 -type f -exec mv {} dest/ ;` for flat moves. For tree moves, `rsync --exclude=node_modules --exclude=.git` is safer.
- **Merging directories with overlapping content** — when merging dir A into dir B, check for filename collisions first. Use `ls A/ | sort > /tmp/a.txt; ls B/ | sort > /tmp/b.txt; comm -12 /tmp/a.txt /tmp/b.txt` to find conflicts. Resolve before copying.
- **Not auditing for dead directories post-merge** — after merging A into B, check if A is now empty (`ls -A A/`) before deleting. If A had subdirs that didn't merge cleanly, they'll be orphaned inside A.

## Big Decisions — When to Delete vs Archive vs Keep

This skill is not just about moving files. It's about making **philosophical decisions** on what deserves to exist.

### Decision Framework

For each directory/project, ask these questions in order:

1. **Has this been touched in 30+ days?** If yes, it's stale. Either archive or delete.
2. **Does it have a clear owner/purpose?** If the purpose is unclear after reading README/PLAN, it's dead.
3. **Does it duplicate another directory?** Merge into the more complete one.
4. **Is it an upstream clone with no local changes?** Delete or replace with git submodule/shallow clone.
5. **Is it a full project with its own GitHub remote?** Move it OUT of the workspace — it doesn't belong nested in another repo.
6. **Is it > 100MB with no recent activity?** Flag for manual review — likely deletable.

### The Workspace Boundary Rule

**The workspace is not a project graveyard.** It's a working surface for:
- Agent configuration (AGENTS.md, SOUL.md, etc.)
- Daily logs and memory
- Active scripts and tooling
- Documentation that doesn't belong to any single project

**Projects with their own repo** (gto-wizard-clone, hermes-guide, open-lovable) should live at `~/project-name/`, not `/workspace/project-name/`. The workspace repo should NOT track them — they have their own remotes.

### Size Budgets

- Workspace root: ≤ 15 real files, ≤ 25 directories
- No single directory > 500MB without explicit justification
- Total workspace: < 5GB (excluding `~/` projects)
- Archive anything completed that's not actively maintained

## Verification

```bash
find /workspace -maxdepth 1 -type f ! -name '.*' | wc -l   # target: ≤ 15
find /workspace -maxdepth 1 -type l | wc -l                  # target: ≤ 5
find /workspace -maxdepth 1 -type l ! -exec test -e {} \\; -print  # should be empty
```
