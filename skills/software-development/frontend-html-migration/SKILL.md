# Frontend HTML to React Migration

Convert standalone HTML files (generated by design tools, specialized models, or static references) into production Next.js page components.

## When to Use

- A specialist model returns standalone HTML/CSS/JS files for new pages
- User provides reference HTML files with the instruction "adapt this"
- You need to integrate external designs into an existing Next.js app
- Converting static prototypes into interactive React pages

## Pipeline

### Step 1: Ingest the HTML

Read the full file. Extract:

- **`:root` CSS variables** — these define the design system. Map them to your app's existing variables or keep them as-is
- **Layout structure** — `<div class="app">`, `<main>`, grids, sidebar vs single-column
- **Component hierarchy** — topbar, config bar, left panel, right panel, bottom section
- **State variables** — look at `let` / `const` declarations at the top of `<script>` blocks
- **Event handlers** — `onclick`, `onchange`, `onmouseenter` — these become React callbacks
- **DOM queries** — `document.getElementById('foo')` — these become `useRef` or `useState`
- **Engine/algorithm code** — Monte Carlo sims, hand evaluators, deck shufflers, data parsers — port these verbatim as TypeScript utility functions

### Step 2: State Mapping Rules

| Vanilla JS | React |
|------------|-------|
| `let state = { x: 1, y: 2 }` | `const [state, setState] = useState(...)` or individual `useState` per field |
| `document.getElementById('x')` | `const xRef = useRef(null)` or `const [x, setX] = useState(initial)` |
| `el.innerHTML = '<div>...'` | JSX return or `setState` with child components |
| `el.addEventListener('click', fn)` | `onClick={fn}` |
| `el.classList.add('foo')` | Conditional class strings or inline style objects |
| `setInterval(fn, ms)` | `useEffect` with interval + cleanup |
| `localStorage.getItem/setItem` | `useEffect` to sync + `useState(() => JSON.parse(...))` initializer |

### Step 3: Preserve Game Engines Verbatim

For poker hand evaluators, Monte Carlo simulators, deck shufflers, and other computation-heavy JS:

- Copy functions directly into the page file or a `lib/` utility file
- Add TypeScript types for input/output interfaces
- Do NOT "optimize" or restructure the algorithm — it's already correct
- Monte Carlo loops: wrap in `setTimeout(() => {...}, 50)` to let UI breathe, with `useRef(cancelled)` for cleanup

### Step 4: Styling Strategy

Use **inline style objects** (not Tailwind utility classes) for reference-adapted pages:

```typescript
<div style={{
  background: '#0b0d0f',
  border: '1px solid #252b32',
  borderRadius: 8,
  padding: '12px 14px',
}}>
```

Reasons: exact matches to reference CSS, no Tailwind config changes, easy to compare with source HTML, avoids library conflicts.

Add responsive breakpoints to `globals.css` as grid-class overrides:

```css
@media (max-width: 1100px) {
  .my-grid { grid-template-columns: 1fr !important; }
}
```

### Step 5: API Integration

- Provide realistic mock data as fallback (arrays of mock objects, not empty arrays)
- Fetch API data with `try/catch` — on failure, use mocks
- Pattern:
  ```typescript
  try {
    const res = await fetch(`${API_BASE}/endpoint`)
    if (!res.ok) throw new Error('API error')
    setData(await res.json())
  } catch {
    setData(MOCK_DATA)
  }
  ```

#### Step 6: Visual Fidelity Check

Before declaring a page complete, verify it against the reference:

1. **Title tag** — `curl -sL http://localhost:PORT/ | grep -o '<title>[^<]*</title>'` must match expected app name
2. **Nav tabs all resolve** — every nav link must return HTTP 200, not 404
3. **Design parity** — the page's accent colors, spacing, border radii, and layout grid must match the reference HTML, not just be "close enough"
4. **Responsive breakpoints** — reference CSS `@media` queries must be ported to `globals.css` as grid class overrides, not left out

**CRITICAL — "looks worse than reference" is a deployment blocker.** The user will reject it. Verify visual fidelity before deploying. If you can't achieve parity, flag it rather than deploying a downgrade.

### Multi-Page Integration Pattern

When integrating multiple generated HTML files (e.g., 5+ pages from a model):

1. Build all pages as `page.tsx` files first
2. Add ALL responsive breakpoints to `globals.css` in one shot
3. Verify ALL nav tabs resolve to real pages (`curl` each one)
4. Build and deploy ONCE, not one-at-a-time
5. Check the live URL for each new page

This avoids repeated `build → kill → restart → verify` cycles for each page.

## Step 7: Design System Translation

Generated HTML often uses a different accent color and CSS variable set than your target app. You MUST translate, not paste raw:

| Source (generated) | Target (this app) | Action |
|---|---|---|
| `#00a98f` teal accent | `#00C853` green | Use target colors everywhere |
| `--panel: #14171b` | `--panel: #1C1C1C` | Map to target equivalent |
| `--bg: #0b0d0f` | `--bg: #0E0E0E` | Prefer target CSS variables |

**The target app's design system is authoritative.** If the generated HTML uses different accent colors, border radii, or spacing, convert them. The user will notice inconsistent accent colors across pages.

### When Layout Grids Differ

If the new page introduces a grid pattern not yet in `globals.css`, add it with a descriptive class name matching the page role: `.icm-pro-grid`, `.play-grid`, `.practice-grid`. Add the responsive override in the same edit.

## Step 8: Pitfalls

- **Process group kill kills the current shell** — `kill -9 -PGID` can kill the CURRENT SHELL (exit code -9). Always run in its own short `terminal()` call, NOT combined with other operations or follow-up `curl`/`echo`. The shell dies before subsequent commands execute.
- **Responsive layouts** — the reference HTML likely has `@media` queries. Port them to `globals.css` immediately — inline `@media` in Next.js components requires `styled-jsx` or CSS modules.
- **Canvas charts** — if the reference uses canvas 2D for charts (donut, bar, etc.), keep canvas rendering with `useRef` + `useEffect`. Set explicit `width`/`height` on the canvas element. Handle responsive sizing with CSS `max-width: 100%` on the wrapper. Do NOT delete the canvas rendering — it works and looks correct. Recharts migration can be a follow-up.
- **Local storage persistence** — if the reference persists state to `localStorage`, keep that pattern. Initialize `useState` with a function that reads `localStorage` so data survives refresh: `useState(() => JSON.parse(localStorage.getItem('key') || '{}'))`
- **Game engine preservation** — Monte Carlo sims, hand evaluators, and deck shufflers are the most valuable code in the reference. Port them verbatim to TypeScript utility functions. Do NOT restructure or "optimize" them — they're already correct.
- **`next.config.ts` rewrite port default** — Generated pages often hardcode an old API port (e.g., `localhost:8002`). Before rebuilding, check that all rewrite destinations in `next.config.ts` point to the current API port. The default is stale on first deploy.
- **Tunnel ingress simplification** — If the tunnel's ingress has separate rules for frontend and API paths, simplify to a SINGLE ingress rule pointing to the frontend. Use Next.js rewrites (`next.config.ts`) to proxy `/api/*` to the backend. This is more reliable than tunnel-level path routing, which can return cryptic 500 errors.
