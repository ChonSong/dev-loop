## Conversion Approach
Each HTML file was a self-contained page with inline CSS (no Tailwind), vanilla JS state management, and DOM manipulation. Conversion pattern:

1. **Read HTML structure** ‚Äî identify layout grids, component hierarchy
2. **Extract CSS** ‚Äî copy `:root` variables, preserve all class-based styles as inline style objects
3. **Convert JS to React state** ‚Äî `let state = {...}` ‚Üí `useState`, `getElementById('x')` ‚Üí `useRef` / `useState`
4. **Replace `innerHTML`** with JSX rendering
5. **Keep game engines** ‚Äî Monte Carlo, hand evaluators, deck shufflers stay as-is

## User Preference ‚Äî Imported Design Quality Standard

The user expects frontend pages to match reference HTML designs with exact visual fidelity. Key preferences:
- Dark theme: `#0b0d0f` background (not Tailwind gray tones)
- Accent: `#00a98f` teal (not #00C853 green from initial deployment)
- Inline styles preferred over Tailwind for reference-adapted pages
- Single-file `'use client'` page components preferred over split files for new pages
- Mock data must be provided when API endpoints aren't ready ‚Äî never show blank/error states
- Generated HTML from specialized models is the starting point; manual refinement follows
