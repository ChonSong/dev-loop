# Solver Preflop Range Architecture (June 2026)

The study page's preflop-range endpoint was rewritten from a hardcoded heuristic to a real solver-backed data model. This reference documents the architecture.

## Data Flow

```
Study Page (React) → POST /api/v1/solver/preflop-range
                           ↓
                    _generate_range(position, stack_depth)
                           ↓
              ┌────────────┼────────────┐
              ↓            ↓            ↓
        Shallow      Deep        Fallback
        (≤60bb)      (>60bb)     (any)
              ↓            ↓            ↓
      Push/fold    Equity-based    Hardcoded
      charts       GTO ranges     (removed)
```

## Three Data Sources

### 1. Precomputed Preflop Equities (all depths)

File: `apps/api/data/preflop_equities.json`

All 169 hands' equity against a random hand. Values sourced from known poker math (Monte Carlo with sufficient samples). Examples:

| Hand | Equity |
|------|--------|
| AA   | 85.2%  |
| KK   | 82.2%  |
| AKs  | 67.0%  |
| 32o  | 40.0%  |
| 22   | 54.5%  |

Generated by: `apps/api/data/preflop_data.py` — writes the static cache. No runtime computation needed.

### 2. Push/Fold Charts (shallow stacks ≤ 60bb)

Location: `apps/solver/strategy/charts/push_*_*.json`

Precomputed GTO solutions for push/fold scenarios. Available for 6 positions × 5 stack depths (10, 20, 40, 60, 100bb). Each chart maps every hand to either `"push"` or `"fold"`.

**Loading**: `_load_chart(position, stack_depth)` finds the nearest available stack depth.

### 3. GTO Equity-Based Model (deep stacks > 60bb)

If no push/fold chart matches, the system sorts all 169 hands by equity (descending) and assigns actions based on standard RFI range widths:

| Position | Raise Width | Typical Hands |
|----------|-------------|--------------|
| UTG      | 15.5%       | 26 hands — premium pairs, big aces |
| HJ       | 21%         | 35 hands — adds suited broadways |
| CO       | 28%         | 47 hands — adds more broadways, small pairs |
| BTN      | 42%         | 70 hands — loosest, adds suited connectors |
| SB       | 45%         | 76 hands — wide range, smaller raise size |
| BB       | 0% (defend) | BB doesn't RFI; defends with ~50% of hands |

Source: `_PREFLOP_RANGES` dict in `apps/api/routers/solver.py`.

## Action Response Format

Each hand in the response includes:

```json
{
  "hand": "AA",
  "action": "raise_2.5bb",   // or "fold", "call", "all_in"
  "frequency": 1.0,           // 0.0 to 1.0 — fold hands always 1.0 now
  "equity": 0.852             // preflop equity vs random
}
```

FT: The `frequency` field was originally 0.0 for fold hands (causing confusing "FOLD — 0%" in the UI). Fixed: fold hands now return `frequency: 1.0` to indicate 100% fold frequency.

## Postflop Solving

The MCCFR engine at `apps/solver/cfr/engine.py` can solve river spots in real-time. Takes ~1s for 50 iterations.
Endpoint: `POST /api/v1/solver/solve`

The solver engine requires `phevaluator` (installed via `uv pip install phevaluator`).

## Common Pitfalls

- **phevaluator not installed**: The solver endpoint degrades gracefully but equities are uncomputable. Fix: `uv pip install phevaluator`. Added to `pyproject.toml` `[dependency-groups] runtime`.
- **Static cache stale**: If `preflop_equities.json` is missing, equity defaults to 0.5. Regenerate by running `apps/api/data/preflop_data.py` under the project venv.
- **Turbo cache hides new routes**: After adding new pages like `/variants`, clear `apps/web/.next` and `apps/web/.turbo` before building.
