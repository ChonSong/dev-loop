- **SMB service:** guest=nobody, workgroup=WORKGROUP, no SMB1, encryption=DEFAULT
- **SMB options set:** map to guest = Bad User + server signing = disabled
- **Created user:** shared (uid 3000, SMB enabled, password: shared1, home: /var/empty)

## WebSocket API Protocol Detail

The TrueNAS middleware WebSocket uses a custom JSON protocol:

```
connect: {"msg": "connect", "version": "1", "support": ["1"]}
response: {"msg": "connected", "session": "uuid"}
rpc: {"id": "1", "msg": "method", "method": "...", "params": [...]}
result: {"id": "1", "msg": "result", "result": ...}
```

CRITICAL: version and support values must be **strings** ("1"), not integers (1).
An integer silently fails with {"msg": "failed"} with no clue why.

Python implementation at /tmp/tn_ws.py provides a reusable call(method, params) function.

## Disk Space Crisis

### Root Cause

The TrueNAS VM has a separate 40 GB boot disk (sdb). When the root filesystem
fills up (tmpfs /run from too many API login attempts), PAM's pututline() fails
with "No space left on device". This prevents ALL authentication including API key.

### Fix

Reboot clears tmpfs. If it's the boot-pool ZFS datasets that are full, check:
```bash
zpool list
zfs list -t all -o name,used,available,quota,referenced
```

Remove quotas on boot datasets if needed:
```bash
zfs set quota=none boot-pool/ROOT/25.10.3.1/root
```

### The pututline() Error

This comes from PAM writing a login record to /var/run/utmp (tmpfs). When
tmpfs is full, authentication fails with a misleading "credentials failed" error.
The credentials are valid -- the system just can't write the audit trail.

## TrueNAS SCALE Container Runtime (25.10.x)

TrueNAS SCALE 25.10 uses **Kubernetes (k3s)** for Apps, not raw Docker:
- app.available -- lists installable apps from TrueNAS catalog
- app.query -- lists installed apps
- No docker.* or kubernetes.* middleware methods available
- Custom apps use IX-Charts (Helm-based), not Docker Compose
- DOCKER_READ/DOCKER_WRITE roles exist in permission system but no Docker API

This means deploying Hermes on TrueNAS requires either:
1. IX-Chart/Helm custom app through the Apps system
2. Docker installed manually on the TrueNAS host (risks breaking on updates)
3. Running Hermes on the existing host with TrueNAS providing storage

## SMB Guest Access - NFSv4 ACL Format

In TrueNAS SCALE 25.x, guest access is filesystem-ACL based, not per-share.

- guestok does NOT exist in the API (Extra inputs are not permitted)
- SMB service guest=nobody by default
- Must set smb_options: map to guest = Bad User
- Must set ACL to grant everyone@ access
- Must restart SMB service after config changes

### Python smbprotocol Limitations

- Empty username/password -> SPNEGO error (anonymous not supported)
- Guest username with empty password -> authenticates but fails on signing
- Named user with password -> works reliably

The error "SMB encryption or signing was required but session was authenticated
as a guest" occurs because TrueNAS requires SMB signing by default. Setting
server signing = disabled in smb_options may not take effect on SCALE 25.x.

## ACL Format Reference

| Tag | Meaning |
|-----|---------|
| owner@ | File owner |
| group@ | File group |
| everyone@ | All users including nobody/guest |
| USER | Specific user (requires who field) |
| GROUP | Specific group (requires who field) |

Permission levels: FULL_CONTROL, MODIFY, READ, WRITE
Flags: INHERIT, NO_PROPAGATE_INHERIT, INHERIT_ONLY

## TrueNAS REST API Notes

- POST with single string param: -d '"/mnt/p1/public"'
- POST with named params: -d '{"path": "/mnt/..."}'
- Job-based endpoints return integer job ID
- Poll jobs at /api/v2.0/core/get_jobs/?id=N
- API key in Authorization header gets redacted in Hermes terminal output
- PUT on /api/v2.0/smb returns 500 on SCALE 25.x - use WebSocket instead

## Key MCP Tool Gaps (not in truenas-mcp v0.2.0)

- user.create / user.update
- smb.update / sharing.smb.update
- filesystem.setacl / filesystem.getacl
- service.restart / service.start / service.stop

Use direct WebSocket API for these operations.
