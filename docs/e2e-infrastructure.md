# E2E Test Infrastructure

The dev loop supports three test layers. Layer 3 (E2E) is auto-generated by Coach via the `coach-test-generator` skill and is validated every review cycle.

## Three Test Layers

```
┌─────────────────────────────────────────────────────────┐
│                    TEST LAYERS                          │
│                                                         │
│  Layer 1: API Tests (pytest)                            │
│  └─ Backend business logic, database queries            │
│  └─ Fast: <5s, runs every Player tick                   │
│  └─ Location: apps/api/tests/                           │
│  └─ Runner: cd apps/api && uv run pytest -q --tb=short  │
│                                                         │
│  Layer 2: Frontend Unit Tests (vitest)                  │
│  └─ Component rendering, state management               │
│  └─ Fast: <10s, runs every Player tick                  │
│  └─ Location: apps/web/src/**/*.test.ts                 │
│  └─ Runner: cd apps/web && npx vitest run               │
│                                                         │
│  Layer 3: E2E Browser Tests (Playwright)                │
│  └─ Full user workflows via POM + spec files            │
│  └─ Slow: 30-120s, runs every Coach tick               │
│  └─ Location: apps/web/e2e/                             │
│  └─ Runner: cd apps/web && npm run test:e2e             │
│  └─ Generated by: coach-test-generator skill            │
└─────────────────────────────────────────────────────────┘
```

## How E2E Tests Are Created

Coach uses a 4-phase protocol (defined in `coach-test-generator` skill) to auto-generate POM + playwright specs for any page:

```
Phase 1: EXPLORE
  └─ Navigate page, snapshot accessibility tree
  └─ Catalog all interactive elements → tests/catalog/{page}.json

Phase 2: ABSTRACT
  └─ Group elements by behavioral pattern
  └─ Detect selector strategy (aria-label vs CSS class vs text)
  └─ Deduce the primary user journey

Phase 3: GENERATE
  └─ Write POM to apps/web/e2e/pages/{PageName}.ts
  └─ Write spec to apps/web/e2e/{page}.spec.ts
  └─ Write reference copies to tests/pom/ + tests/workflows/

Phase 4: VERIFY (every subsequent Coach review)
  └─ Run: cd apps/web && npm run test:e2e
  └─ Check pass/fail, annotate known bugs
  └─ If POM selectors drift: re-run EXPLORE → update POM
  └─ If app bug found: seed SIE learning entry
```

## File Layout Standard

Per industry consensus (Next.js official example, Playwright docs, 2026 best practices):

```
# Monorepo pattern (gto-wizard-clone)
project/
├── apps/web/
│   └── e2e/
│       ├── pages/
│       │   ├── StudyPage.ts          # POM with @playwright/test Page import
│       │   └── PracticePage.ts
│       ├── study.spec.ts             # Playwright test spec
│       ├── practice.spec.ts
│       └── playwright.config.ts      # Config at e2e level (monorepo)
├── tests/
│   ├── pom/                          # Reference POM copies (Coach-readable)
│   ├── workflows/                    # Reference workflow definitions
│   └── catalog/                      # Phase 1 element catalogs

# Single-project pattern (polytopia-clone)
project/
├── e2e/
│   ├── pages/
│   │   └── GamePage.ts
│   └── *.spec.ts
├── playwright.config.ts              # Config at root (single project)
├── tests/
│   ├── pom/
│   ├── workflows/
│   └── catalog/
```

## Agent Responsibilities

| Agent | E2E Responsibility |
|-------|-------------------|
| **Player** | Before coding: check if the target page has e2e tests. If yes, update POM + spec as part of the implementation plan. If adding a new interactive element, add POM method + test step. |
| **Coach** | Every review: run `npm run test:e2e`. Report pass/fail counts in verdict. If tests fail with a new regression, escalate to FIX/REVERT. If tests fail with a known bug, annotate but don't block. If a pattern recurs across projects (e.g., "POM selector drift"), seed a `.learnings/` entry. |
| **SIE Phase 0** | Every 48h: scan projects for e2e infrastructure existence. Flags: "No e2e tests at all," "E2E tests fail," "POM selector drift >20%." Seeds `.learnings/` entries for uncovered gaps. |

## POM Design Rules

Generated POMs follow these rules (enforced by `coach-test-generator` skill):

1. **Stable selectors only**: `aria-label`, `role`, data attributes, or text content match. Never CSS classes or nth-child.
2. **One method = one action**: `selectPosition()`, not `interactWithPosition()`.
3. **Import `Page` from `@playwright/test`** — never `any`. Type safety forces standard Playwright API.
4. **Section visibility for state detection**: For pages without `aria-pressed`, use conditional section rendering as state indicator ("is Timer Duration visible?" → must be Timed Drill mode).
5. **Throw on failure, return on success**: Assertions throw `Error`. Queries return `null` or typed data.

## Known Bug Tracking

When e2e tests fail due to a known application bug (not a test infrastructure issue), annotate the test rather than skipping it:

```typescript
// In the spec file:
try {
  await study.assertGTOActionsSumTo100(actions, 10);
} catch (e: any) {
  test.info().annotations.push({
    type: 'known-bug',
    description: `GTO frequency bug: ${e.message}`
  });
}
```

Known bugs are still reported in the Coach verdict but don't block APPROVE. When the bug is fixed, the annotation catches it — the test starts passing, and Coach notes the improvement.

## SIE Integration

When SIE Phase 0 scans for coverage blind spots, it checks:

| E2E Check | How | Flags If |
|-----------|-----|----------|
| Infrastructure exists | `ls apps/web/e2e/*.spec.ts` | No browser tests at all |
| Tests pass | `cd apps/web && npm run test:e2e` | Tests fail (regression) |
| POM freshness | Compare catalog element count vs live DOM | >20% drift since catalog was written |

If a project fails any of these checks, SIE seeds a `.learnings/` entry that its Phase 1 pipeline picks up. On Recurrence-Count >= 3, the pattern promotes to skill authoring.

## Cron Commands Reference

```bash
# Run all e2e tests
cd apps/web && npm run test:e2e

# Run specific page tests
cd apps/web && npm run test:e2e -- e2e/study.spec.ts
cd apps/web && npm run test:e2e -- e2e/practice.spec.ts

# Run with HTML report
cd apps/web && npm run test:e2e -- --reporter=html

# Update Playwright browsers (if needed)
cd apps/web/e2e && npx playwright install chromium
```
